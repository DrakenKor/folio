Logs usually start as notes left by the program.

One line when a request starts. One line when it fails. A few values placed into a string because someone might need them later.

That works for one process on one host. It becomes weaker as soon as there are many services, many hosts, queues, retries, and deploy versions. At that point the log is not just text. It is data created by the running system.

The useful question is:

```text
Could we query this record during an incident without already knowing the answer?
```

If the answer is no, the log may still be comforting. It is just not very operational.

## Start with the record

A log line is a record. The message is one field in that record.

Useful fields tend to be boring:

| Field | Why it matters |
| --- | --- |
| `timestamp` | Orders events approximately and supports time windows |
| `level` | Separates expected information from operator action |
| `event` | Names the fact that happened |
| `service` | Identifies the component that emitted the record |
| `environment` | Separates production, staging, and development |
| `version` | Connects behavior to a deployed build |
| `correlation_id` | Follows one request or workflow across boundaries |
| `actor_id` | Identifies the user, service, or job taking the action |
| `aggregate_id` | Identifies the domain object being changed |
| `attempt` | Shows retries without guessing from repeated messages |
| `duration_ms` | Makes latency queryable |
| `result` | Makes success, failure, timeout, and skipped paths explicit |
| `error_class` | Groups failures by type instead of by prose |

The fields are the interface. If the only stable part of a log is the sentence, querying it becomes string archaeology.

<Callout title="Rule of thumb">
  Write logs as records first and messages second. Put anything you will filter, group, alert on, or join across services into a named field.
</Callout>

## Messages are not enough

This is easy to write:

```python
logger.info(f"User {user_id} created post {post_id} in {duration_ms}ms")
```

It is also easy to read locally. The problem appears later. To count post creation latency by service version, a log system now has to parse a sentence that was written for a human.

This shape is quieter and more useful:

```python
logger.info(
    "post_created",
    extra={
        "event": "post.created",
        "service": "posts-api",
        "version": app_version,
        "environment": environment,
        "correlation_id": request.headers.get("X-Request-ID"),
        "user_id": user_id,
        "post_id": post_id,
        "content_length": len(content),
        "duration_ms": duration_ms,
        "result": "success",
    },
)
```

The sentence can still exist. It should not be the only place the data lives.

## Event names

An event name should be stable enough to query for months.

Good event names are usually:

1. Specific.
2. Lowercase.
3. Past tense for facts.
4. Consistent across services.
5. Not packed with variable data.

For example:

| Better | Weaker |
| --- | --- |
| `post.created` | `created post 928173` |
| `payment.authorization_failed` | `payment bad` |
| `job.retry_scheduled` | `retrying maybe` |
| `cache.stampede_blocked` | `lock worked` |
| `request.deadline_exceeded` | `too slow` |

The event name should answer "what kind of thing happened?" The fields should answer "to which object, under which request, with which result?"

## Correlation IDs

A correlation ID gives one logical operation one name as it crosses services, queues, workers, and storage systems.

At the edge, generate one if the caller did not send one:

```python
def get_correlation_id(request):
    incoming = request.headers.get("X-Request-ID")
    if incoming:
        return incoming[:128]

    return uuid.uuid4().hex
```

Then pass it through every boundary:

```python
def create_post(request):
    correlation_id = get_correlation_id(request)

    post = db.insert_post(
        user_id=request.user.id,
        content=request.json["content"],
    )

    queue.publish(
        "post.created",
        {
            "post_id": post.id,
            "user_id": request.user.id,
            "correlation_id": correlation_id,
        },
    )

    logger.info(
        "post_created",
        extra={
            "event": "post.created",
            "post_id": post.id,
            "user_id": request.user.id,
            "correlation_id": correlation_id,
        },
    )

    return {"post_id": post.id}
```

Without the ID, each service has a local story. With the ID, the workflow has a shared thread.

## Logs, metrics, and traces

These three tools overlap, but they are not the same thing.

| Tool | Shape | Good for |
| --- | --- | --- |
| Log | Event record | Explaining what happened in one case |
| Metric | Numeric time series | Watching rates, latency, saturation, and SLOs |
| Trace | Causal span graph | Following one request through many services |

A useful incident often needs all three:

1. Metric shows `payment.authorization_failed` increased.
2. Trace shows failures cluster behind one dependency call.
3. Logs show the dependency returned `429` after a rollout changed request volume.

The log gives detail. The metric gives scope. The trace gives shape.

## Levels are an operating contract

Log levels should mean something stable.

| Level | Meaning | Example |
| --- | --- | --- |
| `ERROR` | Something failed and likely needs action or automatic recovery | Payment authorization exhausted retries |
| `WARN` | Something unusual happened, but the system handled it | Cache refresh skipped and stale value served |
| `INFO` | A useful business or lifecycle fact happened | Post created, job completed, deploy started |
| `DEBUG` | Detail useful during development or narrow investigation | Intermediate parser state |

`ERROR` should not mean "the code took a branch we expected." If expected validation failures are logged as errors, the alert stream becomes noise. If real failures are logged as info, the system loses visibility exactly when it needs it.

Levels are not decoration. They are routing rules for attention.

## Aggregation is part of the system

Local logs are temporary. Containers restart. Hosts disappear. Disk fills. A service that only logs to local files is observable only while the local file is still reachable.

At scale, logs need a pipeline:

```text
application -> stdout/file -> collector -> transport -> index/store -> query/alert
```

Each step has failure modes:

| Step | Failure to plan for |
| --- | --- |
| Application | Blocks while trying to write logs |
| Collector | Drops logs during restart or backpressure |
| Transport | Reorders or duplicates records |
| Index | Rejects records with unexpected field types |
| Store | Retention expires before an investigation starts |
| Query | High-cardinality fields make searches slow or expensive |

This does not mean every app needs a custom logging platform. It means logging is a production dependency, and the dependency should have backpressure, drop behavior, retention, and cost limits that are understood.

## Sampling

Some events are too frequent to keep in full detail.

Page views, health checks, cache hits, and successful background jobs can easily dominate log volume. Keeping every record may cost more than the value it provides.

Sampling should be deliberate:

1. Always keep errors.
2. Always keep security and audit events.
3. Sample high-volume success paths.
4. Sample by correlation ID or stable key, not by individual line.
5. Record the sample rate on the event.

Stable sampling keeps related logs together:

```python
import hashlib


def should_sample(key, rate):
    digest = hashlib.sha256(key.encode("utf-8")).hexdigest()
    bucket = int(digest[:8], 16) / 0xFFFFFFFF
    return bucket < rate


def log_post_view(post_id, user_id, correlation_id):
    if not should_sample(correlation_id, rate=0.01):
        return

    logger.info(
        "post_viewed",
        extra={
            "event": "post.viewed",
            "post_id": post_id,
            "user_id": user_id,
            "correlation_id": correlation_id,
            "sample_rate": 0.01,
        },
    )
```

Sampling by line can leave half a story. Sampling by workflow keeps the story usable when it is selected.

<Callout type="warning" title="Warning">
  Do not sample correctness, security, billing, or audit facts unless another durable system already stores them with the required guarantees.
</Callout>

## Retention

Retention is part of the design, not just a storage setting.

Different records need different lifetimes:

| Record type | Typical retention thought |
| --- | --- |
| Debug logs | Short, often hours or days |
| Request logs | Long enough for incident review and support |
| Error logs | Long enough to compare releases and recurring failures |
| Security logs | Long enough for investigation and policy |
| Audit events | Long enough for product, legal, or financial requirements |

Hot storage is fast and expensive. Cold storage is slower and cheaper. The retention policy should match the way the data is used.

If support tickets arrive two weeks after the user action, seven days of searchable logs may be too short. If debug logs contain large payloads, keeping them for a year is usually waste.

## Logs as audit trail

Some logs are operational. Some are audit records. They should not quietly share the same standards.

An audit trail usually needs stricter properties:

1. Stable schema.
2. Append-only storage.
3. Actor and target identifiers.
4. Before and after values when appropriate.
5. Authorization context.
6. Tamper evidence or restricted write access.
7. Clear retention.
8. PII minimization.

For example:

```python
def change_role(actor_id, user_id, new_role):
    old_role = db.get_user_role(user_id)

    db.update_user_role(user_id, new_role)

    audit_log.write(
        {
            "event": "user.role_changed",
            "actor_id": actor_id,
            "user_id": user_id,
            "old_role": old_role,
            "new_role": new_role,
            "result": "success",
        }
    )
```

This is not the same as a debug line that says "updated user." It is a domain record. It should be treated like one.

## Event streams and logs

Logs can look like event streams because both are append-only records.

The difference is the contract.

| Question | Debug log | Event stream |
| --- | --- | --- |
| Is delivery required? | Usually no | Usually yes |
| Can consumers depend on schema? | Often weak | Should be strong |
| Can the record be replayed? | Maybe | Usually yes |
| Is ordering defined? | Often no | Usually per key or partition |
| Is retention a product concern? | Sometimes | Often yes |

It is fine to use the same event shape for both observability and downstream work. It is risky to let an accidental debug log become the source of truth for a workflow.

If another system must react to the fact, write an event with a delivery contract:

```python
def publish_post_created(tx, post):
    tx.insert_outbox_event(
        event_type="post.created",
        aggregate_id=post.id,
        payload={
            "post_id": post.id,
            "user_id": post.user_id,
            "created_at": post.created_at.isoformat(),
        },
    )
```

The logger can record that the event was written. The outbox is the delivery boundary.

## PII and secrets

Logs spread quickly. They move from application memory into collectors, indexes, dashboards, tickets, notebooks, and cold storage. Anything placed into a log is likely to be copied.

So logging needs a negative list and a positive list.

Do not log:

1. Passwords.
2. Session tokens.
3. API keys.
4. Full payment details.
5. Raw authorization headers.
6. Large request bodies by default.
7. Private messages unless the feature explicitly requires secure audit handling.

Do log identifiers that allow lookup in the proper system:

```python
logger.info(
    "payment_authorization_failed",
    extra={
        "event": "payment.authorization_failed",
        "payment_id": payment.id,
        "order_id": order.id,
        "gateway": "stripe",
        "gateway_error_code": error.code,
        "correlation_id": correlation_id,
    },
)
```

The log should point to sensitive data. It should rarely contain it.

## Schema drift

Structured logs fail quietly when field names drift.

One service logs `user_id`. Another logs `uid`. A third logs `userId`. One version writes `duration_ms` as a number, another as a string. Queries become partial, dashboards lie by omission, and incident review gets slower.

A small schema contract helps:

```python
def log_event(logger, event, **fields):
    required = {
        "event": event,
        "service": SERVICE_NAME,
        "version": VERSION,
        "environment": ENVIRONMENT,
    }

    logger.info(event, extra={**required, **fields})
```

For larger systems, this becomes a shared logging package or schema registry. The point is not ceremony. The point is to make the important fields boring and consistent.

## What I notice

The systems that are easiest to debug do not necessarily log the most.

They log the same few facts consistently. They use stable event names. They pass correlation IDs without making it optional. They separate expected validation from real failure. They keep the fields small enough to index. They know which records are sampled and which are never sampled.

The difficult systems often have many logs but few records. The messages are expressive. The data is hard to query.

That distinction shows up during incidents. A good log record removes guesses. It does not remove all work, but it narrows the work.

## Design the log event

For a new user-facing write, define the log event before the incident.

Example for post creation:

| Field | Value |
| --- | --- |
| `event` | `post.created` |
| `level` | `INFO` |
| `service` | `posts-api` |
| `correlation_id` | Generated at edge, propagated to queue event |
| `user_id` | Author id |
| `post_id` | Created post id |
| `content_length` | Integer, not content body |
| `duration_ms` | API handler duration |
| `result` | `success`, `validation_failed`, `timeout`, `unknown` |
| `attempt` | Request attempt number if retry-aware |
| `sample_rate` | `1.0` for writes |

Then define the queries that should work:

```text
event:post.created result:success service:posts-api
event:post.created user_id:1234
event:post.created correlation_id:abc123
event:post.created result:timeout version:2026.05.23.1
```

If the queries cannot be answered, the event is not shaped yet.

## Metrics that help

Useful logging metrics are about the pipeline and the records:

| Metric | Why it matters |
| --- | --- |
| Log events emitted per service | Shows volume and cost shape |
| Log events dropped by collector | Shows observability loss |
| Error logs by event name | Shows failure category without string parsing |
| Unknown or timeout results | Shows work that needs reconciliation |
| Missing correlation IDs | Shows propagation gaps |
| Index rejection count | Shows schema drift or bad field types |
| Query latency | Shows whether logs are still usable during incidents |
| Storage by event type | Shows which events dominate cost |
| Sampled event rate | Shows whether sampling is behaving as expected |

The logging system itself needs observation. Otherwise it can fail at the same time the application needs it most.

## What I think

The practical sequence is:

1. Define common fields.
2. Use stable event names.
3. Pass a correlation ID through every boundary.
4. Put queryable values in fields, not prose.
5. Keep secrets and large payloads out by default.
6. Treat log levels as an operating contract.
7. Aggregate logs outside the host or container.
8. Sample high-volume success paths by workflow.
9. Never sample records that carry correctness, security, billing, or audit meaning unless another durable source exists.
10. Set retention by record type.
11. Watch the logging pipeline for drops and schema rejects.

The underengineering move is to log fewer things with better shape.

A small set of stable records is more useful than a large pile of sentences.

## Tutorial checklist

For any important event, fill this out:

| Question | Example answer |
| --- | --- |
| Event name | `payment.authorization_failed` |
| Level | `ERROR` if retries exhausted, `WARN` if handled by retry |
| Service | `checkout-api` |
| Correlation ID | Required from edge request or generated at ingress |
| Actor | `user_id` or calling service |
| Aggregate | `order_id`, `payment_id` |
| Result values | `success`, `declined`, `timeout`, `unknown`, `retry_scheduled` |
| Duration field | `duration_ms` as a number |
| Error grouping | `error_class`, `gateway_error_code` |
| Sampling | Never sampled for payment state changes |
| Retention | Hot 30 days, cold 1 year |
| Sensitive data | No card data, no raw authorization headers |
| Query to support | Find all failed authorizations for an order |
| Alert to support | Error rate by gateway and deploy version |

Then ask:

```text
Can this record be found, grouped, and trusted without parsing a sentence?
```

If not, the log is not structured enough yet.

## Summary

1. Logs are data produced by the running system.
2. A useful log line is a record, not just a message.
3. Event names should be stable, specific, and queryable.
4. Correlation IDs connect local service stories into one workflow story.
5. Logs, metrics, and traces answer different questions.
6. Log levels route attention and should have consistent meaning.
7. Aggregation, indexing, retention, and drops are part of the logging design.
8. Sampling should preserve workflows and never hide important correctness records.
9. Audit logs need stricter guarantees than debug logs.
10. Logs can feed event thinking, but delivery contracts belong in event streams.
11. The best logs reduce guessing during incidents.
