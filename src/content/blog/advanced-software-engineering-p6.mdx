Consistency bugs usually look small at first.

A user changes their display name and sees the old value on another screen. A follower count goes down and then up. A search result includes a deleted post for a few seconds. An inventory number says one item is left, but checkout fails.

Those cases do not all need the same fix.

The useful question is narrower:

```text
What promise does this read need from the write that came before it?
```

Sometimes the answer is "the latest value, always." Sometimes it is "the user's own write should be visible to them." Sometimes it is "within a few seconds is fine." The architecture should follow that promise, not a general preference for strong or eventual consistency.

## Start with the read promise

Consistency is not one setting for the whole system. It is a promise made by a read path.

Examples:

| Promise | Meaning | Common fit |
| --- | --- | --- |
| Strong consistency | A read observes the latest committed write | Balances, locks, inventory, uniqueness |
| Read-your-writes | The writer sees their own update immediately | Profile edits, drafts, settings |
| Monotonic reads | A user does not go backward after seeing a newer value | Timelines, document versions |
| Bounded staleness | Data can lag by a known amount | Search, feeds, counters |
| Eventual consistency | Replicas or projections converge later | Analytics, rankings, social counts |

It is easy to reach for strong consistency everywhere because it feels safer, or eventual consistency everywhere because it feels scalable.

Both are vague without the product promise behind them.

<Callout title="Rule of thumb">
  Choose consistency per invariant. If a stale read can corrupt money, access, ownership, or uniqueness, coordinate. If it only makes a display temporarily old, set a lag budget.
</Callout>

## The display name bug

A common bug:

```text
1. User updates display name from "Mina" to "Mina K."
2. API writes the leader database.
3. Profile page reads from a replica.
4. Replica is 800ms behind.
5. User sees the old name and retries the update.
```

The write worked. The read promise failed.

There are several fixes, with different costs:

| Fix | Cost | When it fits |
| --- | --- | --- |
| Read from leader after write | More leader load | The writer must see the latest value |
| Sticky session to same replica | More routing state | Replicas can preserve session order |
| Return updated value from write response | Less follow-up read | UI only needs the changed fields |
| Show "saving" until projection catches up | Product accepts pending state | Derived views update asynchronously |
| Wait for replica apply before success | Higher write latency | Read-after-write must work from replicas |

Do not start with "we need stronger consistency." Start with "which read went stale, for whom, and for how long?"

## CAP, but only the useful part

The simplified CAP theorem line is usually "consistency, availability, partition tolerance: pick two."

The practical version is smaller:

```text
When the network is unreliable, should this operation reject work or serve a possibly stale answer?
```

Network partitions are not always dramatic. A replica can be behind. A region can lose a link. A queue consumer can pause. A deployment can route traffic unevenly. In those cases, the system chooses.

For an account balance write, reject or wait.

For a follower count read, serve the stale count.

For a permission check, read the authoritative source or fail closed.

For a recommendation module, return an empty list or cached result.

CAP is useful when it pushes this decision into the design. It is less useful as a slogan.

## Strong consistency is coordination

Strong consistency means the system coordinates before answering.

Coordination has concrete costs:

1. Extra network round trips.
2. More time spent holding locks or transactions open.
3. Lower availability during replica or region problems.
4. More pressure on the leader or quorum.
5. More complicated failover behavior.

That does not make strong consistency bad. It makes it expensive enough to reserve for invariants.

For example, moving credits between accounts should coordinate:

```python
def transfer_credits(from_account_id, to_account_id, amount):
    with db.transaction() as tx:
        debited = tx.execute(
            """
            UPDATE accounts
            SET balance = balance - %(amount)s
            WHERE id = %(from_account_id)s
              AND balance >= %(amount)s
            """,
            {
                "from_account_id": from_account_id,
                "amount": amount,
            },
        )

        if debited.row_count != 1:
            raise InsufficientFunds()

        tx.execute(
            """
            UPDATE accounts
            SET balance = balance + %(amount)s
            WHERE id = %(to_account_id)s
            """,
            {
                "to_account_id": to_account_id,
                "amount": amount,
            },
        )

        tx.insert_ledger_entry(
            from_account_id=from_account_id,
            to_account_id=to_account_id,
            amount=amount,
        )
```

The transaction is not an implementation detail. It is the consistency boundary.

This path should not read from a stale projection to decide whether money can move. A stale read here is not a UI issue. It can create incorrect state.

## Eventual consistency is a lag budget

Eventual consistency should not mean "we hope it catches up."

It should mean:

```text
This derived value may lag the source of truth by up to N seconds.
If lag exceeds N, the product changes behavior.
```

For a public profile card, the source of truth might be `users`, while search, feed headers, and mention previews update from an event.

```python
def update_profile(user_id, display_name):
    with db.transaction() as tx:
        version = tx.next_profile_version(user_id)
        tx.update_user_profile(
            user_id=user_id,
            display_name=display_name,
            version=version,
        )
        tx.insert_outbox_event(
            event_type="profile.updated",
            aggregate_id=user_id,
            payload={
                "user_id": user_id,
                "version": version,
            },
        )

    return {
        "user_id": user_id,
        "display_name": display_name,
        "version": version,
    }
```

Workers update derived views:

```python
def handle_profile_updated(event):
    user_id = event.payload["user_id"]
    version = event.payload["version"]

    profile = db.get_user_profile(user_id)

    search.upsert_user(
        user_id=user_id,
        display_name=profile.display_name,
        source_version=version,
    )

    cache.delete(f"profile_card:v3:user:{user_id}")
```

Search may lag. The profile edit response does not need to wait for search if the product can tolerate that lag.

The important part is measuring it.

## Read-your-writes without making everything strong

Many product paths only need the writer to see their own latest write. Everyone else can see it later.

A simple pattern is to return a version from the write and carry it into the next read:

```python
def update_settings(user_id, settings):
    version = db.update_settings(
        user_id=user_id,
        settings=settings,
    )

    return {
        "settings": settings,
        "version": version,
    }


def get_settings(user_id, min_version=None):
    if min_version is None:
        return replicas.any().get_settings(user_id)

    replica = replicas.find_one_with_version_at_least(
        user_id=user_id,
        version=min_version,
        timeout_ms=80,
    )

    if replica is not None:
        return replica.get_settings(user_id)

    return db.leader().get_settings(user_id)
```

This keeps normal reads cheap. Only reads that require a known version pay the leader or waiting cost.

That is a useful pattern because most traffic is not immediately after a write.

## Quorums

Quorum systems make the read/write tradeoff explicit.

With `N` replicas:

```text
W = number of replicas that must confirm a write
R = number of replicas that must answer a read
```

If `W + R > N`, the read and write sets overlap. In the simple case, that overlap lets the read observe the latest write.

For `N = 3`:

| W | R | Shape |
| ---: | ---: | --- |
| 1 | 1 | Fast, but stale reads are easy |
| 2 | 1 | Writes wait for a majority, reads can still be stale |
| 1 | 2 | Writes are fast, reads compare more replicas |
| 2 | 2 | Read and write quorums overlap |
| 3 | 1 | Writes are slow, reads are cheap |

Teaching version:

```python
def quorum_intersects(replica_count, write_confirmations, read_confirmations):
    return write_confirmations + read_confirmations > replica_count


assert quorum_intersects(3, 2, 2)
assert not quorum_intersects(3, 1, 1)
```

The real systems are more complicated. Clocks, leader terms, hinted handoff, read repair, partitions, and stale replicas all matter. But the arithmetic is still useful: stronger reads and writes require more agreement, and more agreement costs latency and availability.

## Replication lag is a product metric

If a feature depends on eventual consistency, replica lag is not just an infrastructure metric.

It is part of the product behavior.

Useful metrics:

| Metric | Why it matters |
| --- | --- |
| Replica apply lag | Shows how far followers are behind the leader |
| Projection lag | Shows how far read models are behind events |
| Queue age of oldest event | Shows whether async work is still inside its promise |
| Stale reads served | Shows how often users see old data by design |
| Read fallback to leader | Shows cost of read-your-writes paths |
| Conflict count | Shows when multi-writer assumptions are failing |
| Repair job changes | Shows how often convergence needs help |

Lag should have an SLO if users can observe it:

```text
profile projection p99 lag < 5 seconds
search indexing p95 lag < 30 seconds
analytics dashboard lag < 10 minutes
```

Without a lag target, eventual consistency has no operating boundary.

## Conflicts are design problems

Conflicts happen when two writers can update related state without a single authority.

The easiest conflict strategy is to avoid conflicts:

1. One owner for each entity.
2. One partition key for ordered updates.
3. One writer for a field when possible.
4. Optimistic concurrency for user-edited documents.

Optimistic concurrency is often enough:

```sql
UPDATE documents
SET body = $1,
    version = version + 1,
    updated_at = now()
WHERE id = $2
  AND version = $3;
```

If zero rows update, the client edited an old version. The product can ask them to reload, merge, or choose between versions.

Last-write-wins is acceptable for some low-value fields:

```text
display theme
sort preference
temporary draft title
```

It is a poor default for fields where intent matters:

```text
account balance
inventory reservation
document body
permission grant
seat assignment
```

For counters and sets with multiple writers, use a known conflict-free approach or a database feature that already implements the merge semantics. Hand-rolling CRDTs is rarely where I would start unless the product genuinely needs offline multi-writer edits.

## What I notice

Consistency discussions often mix three different questions:

1. Can the system store the correct fact?
2. Can every reader see that fact immediately?
3. Can the product tolerate some readers seeing an older fact?

Those are separate.

A system can have a strongly consistent source of truth and eventually consistent read models. That is common and practical.

A user profile might be strongly written to `users`, then eventually projected to search, feed headers, notifications, and caches. That does not make the system sloppy. It means each read path has a different promise.

The part that causes bugs is not eventual consistency by itself. The bugs come from forgetting which reads are eventual.

## Make pending states explicit

Async systems create intermediate states. The product should name them.

Examples:

| Operation | Better state model |
| --- | --- |
| Video upload | `uploaded`, `processing`, `ready`, `failed` |
| Search indexing | `saved`, `indexing`, `searchable` |
| Payment | `authorized`, `captured`, `settled`, `failed` |
| Import job | `queued`, `running`, `completed`, `failed` |
| Profile update | `saved`, `visible_in_search_soon` if search lag matters |

This does not mean every UI needs extra labels. It means the backend should not call a projection complete before it is complete.

If the write is accepted but derived work is delayed, the API response should say that.

```json
{
  "status": "saved",
  "search_status": "indexing",
  "profile_version": 42
}
```

Precise state names reduce retry loops, support tickets, and incorrect client assumptions.

## Strong where the invariant lives

A useful split:

| Data | Consistency level I would start with |
| --- | --- |
| Ledger entries | Strong transaction around double-entry invariant |
| Available inventory | Strong reservation or single-writer partition |
| Unique username | Strong uniqueness constraint |
| Permission checks | Strong or fail closed for sensitive actions |
| Idempotency keys | Strong insert/read in the command transaction |
| Profile display name | Strong source of truth, eventual projections |
| Feed item delivery | Eventual, with repair and replay |
| Search index | Eventual, with lag metric |
| View count | Eventual or approximate |
| Analytics dashboard | Eventual, often batched |

Do not make the search index strongly consistent with the profile table unless the product actually needs search to update before the profile edit returns. Usually it does not.

Do not make inventory eventual if the product sells scarce items and overselling is unacceptable. Usually it cannot.

## Repair is part of eventual consistency

Eventual consistency needs a repair path. Otherwise "eventual" means "unless an event was dropped."

Common repair tools:

1. Outbox replay from a durable source.
2. Periodic reconciliation from source of truth to projection.
3. Dead-letter queues for poison events.
4. Version checks on projections.
5. Idempotent handlers that can run more than once.

Example reconciliation:

```python
def repair_profile_projection(batch_size=500):
    stale_profiles = db.find_profiles_newer_than_search(batch_size)

    for profile in stale_profiles:
        search.upsert_user(
            user_id=profile.user_id,
            display_name=profile.display_name,
            source_version=profile.version,
        )
```

The repair job should be boring. It compares source version to projection version and updates missing or old projections.

If a derived system cannot be repaired from the source of truth, it is not just a projection. It is another source of truth, and the consistency story needs to be treated more seriously.

## Multi-region makes the tradeoff visible

Inside one region, strong consistency can be cheap enough for many paths.

Across regions, the cost becomes easier to see.

A write that must coordinate between Tokyo, Virginia, and Frankfurt waits on speed-of-light latency, network variance, and failure handling. If every profile edit needs global agreement before returning, the user pays for that design on every edit.

Common patterns:

| Pattern | Fit |
| --- | --- |
| Single write region | Simpler correctness, higher latency for distant writers |
| Regional reads with one write leader | Good for read-heavy products |
| Per-tenant or per-user home region | Keeps most writes near their owner |
| Active-active with conflict resolution | Useful when local writes must continue during partition |
| Global strong consistency | Reserve for narrow invariants that justify the latency |

The smaller the consistency boundary, the easier multi-region becomes.

## What I think

The practical sequence is:

1. Write down the invariant.
2. Decide which reads must observe it immediately.
3. Put those reads on the coordinated path.
4. Give every derived read model a lag budget.
5. Return versions from writes.
6. Use read-your-writes only for users who need it.
7. Make conflict policy explicit.
8. Measure lag and stale reads.
9. Build repair from the source of truth.

The main underengineering move is not "make everything eventual." It is "make only the invariant strong."

That keeps the expensive guarantee near the data that needs it.

## Tutorial checklist

For any consistency-sensitive feature, fill this out:

| Question | Example answer |
| --- | --- |
| Feature | Profile update |
| Source of truth | `users` table |
| Invariant | User profile version increases monotonically |
| Strong path | Update `users` and outbox in one transaction |
| Eventual paths | Search, feed headers, profile card cache |
| Read-your-writes | Writer reads leader or replica with `version >= returned_version` |
| Stale promise | Other users may see old display name for up to 5 seconds |
| Conflict policy | Last-write-wins for display name, versioned writes for bio |
| Lag metric | p99 profile projection lag |
| Failure behavior | Show saved profile immediately; search catches up later |
| Repair path | Reconcile projection version from `users` to search |
| Alert | p99 projection lag above 5 seconds for 10 minutes |

Then ask:

```text
If a stale read happens here, is it annoying, misleading, or incorrect?
```

Annoying can often be eventual.

Misleading needs product handling.

Incorrect needs coordination.

## Summary

1. Consistency is a read promise, not a single system-wide setting.
2. Strong consistency is coordination, and coordination costs latency and availability.
3. Eventual consistency should have a lag budget and repair path.
4. Many systems use a strong source of truth with eventual read models.
5. Read-your-writes can be targeted with versions and leader fallback.
6. Quorums make read/write agreement explicit, but they do not remove operational complexity.
7. Conflict resolution should follow product semantics; last-write-wins is not a safe default.
8. Pending states should be named when derived work happens later.
9. Measure replica lag, projection lag, stale reads, conflicts, and repair activity.
10. Make the invariant strong, then let lower-risk projections lag intentionally.
